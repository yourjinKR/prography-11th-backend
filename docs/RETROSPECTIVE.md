# 과제 회고 (제출용)


## 가장 큰 설계 고민: Member와 Cohort의 관계
초기에는 `Member` 엔티티에 보증금/공결횟수를 직접 넣는 구조를 생각했습니다.  
하지만 출석/벌금/보증금은 “기수 단위”이 강하고, 스펙도 현재 기수(11기) 기준 동작을 전제로 하고 있어서 `CohortMember` 중간 엔티티를 두는 방식이 더 맞다고 결론냈습니다.

이를 통해 공결 횟수를 기수별로 분리 관리 가능해졌으며 출석/보증금 이력(DepositHistory) 연결이 명확해졌습니다.  
이후 기수가 늘어나도 모델 자체를 변화시킬 필요 없이 확장이 가능해졌습니다.

그러나 쿼리에 `JOIN`절이 추가되는 등 복잡성은 다소 증가했습니다.

## 출석-보증금 로직에서의 실수와 보정
가장 신경 쓴 부분은 “출석 상태 변경 시 보증금 자동 조정”이었습니다.  
스펙에서 요구한 핵심은 단순 재계산이 아니라 `oldPenalty -> newPenalty`의 차이(`diff`)를 반영하는 것이었습니다.

구현 중 놓쳤던 포인트:
- 기수 변경 시 이전 기수 part/team 참조가 남을 수 있는 케이스
- 세션 요약 API에서 `sessionId` 집계 반영이 약했던 부분

보정 방식:
- 정합성 점검 문서로 이슈를 먼저 기록
- 로직 수정 후 전체 테스트 재실행
- 회귀 테스트로 다시 확인

이 과정을 통해 “코드가 동작하는 것”과 “스펙과 정확히 일치하는 것”은 다른 문제라는 점을 다시 체감했습니다.

## 검증 순서가 중요한 API를 다루며 배운 점
QR 출석 체크는 검증 순서 자체가 계약에 가까웠습니다.  
어떤 조건이 먼저 실패하느냐에 따라 에러 코드가 달라지므로, 단순히 예외를 던지는 수준이 아니라 순서를 코드에서 고정해야 했습니다.

정리하면 다음 순서를 의식적으로 유지했습니다.

1. QR 유효성
2. QR 만료
3. 세션 진행 상태
4. 회원 존재/탈퇴
5. 중복 출석
6. 기수 회원 정보 존재

## 최종 테스트에 대한 고민

테스트 코드에 대한 이해도는 아직 부족한 관계로 테스트 사전에 제공받은 `api-spec`과 `Agent.md`를 기반으로 AI agent가 주로 작성했습니다.  

그렇기에 제출 전 최종 테스트 과정은 입력문과 출력문을 직접 작성해가며 Swagger UI에서 테스트를 진행했습니다.  

이 과정에서 다음과 같은 의문점이 들었습니다.

- 결국 최종적으로 사람이 직접 테스트할 수 밖에 없는가?
- 테스팅 과정을 AI에게 완벽하게 맡길 수는 없을까?
